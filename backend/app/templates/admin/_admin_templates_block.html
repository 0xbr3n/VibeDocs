Placeholder guide — how to edit VibeDocs report templates

{# CRITICAL: every literal Jinja example below MUST live inside a `{% raw %}` block. The page itself is a Jinja template, so any raw `{{ ... }}` or `{% ... %}` token would otherwise be interpreted at PAGE-render time — and `{%p ... %}` (docxtpl's paragraph-loop tag) isn't valid Jinja2 syntax, so it crashes the page with a 500. Wrapping the guide content in `raw` turns every brace pair into displayable text. #} {% raw %}

Every report template (the Word .docx files under backend/word_templates/) is rendered through the docxtpl engine before delivery. To customise a template:

💡 New: Auto-Transform — Uploading a stock VibeDocs template (Heading 2 "Detailed Findings" with one example finding section) via ↑ Replace .docx now automatically wraps the structured section with a {%p for f in findings %} loop and fills the CVSS table + SubHeading bodies with the correct {{ f.* }} placeholders. You no longer need to add placeholders by hand inside the example finding section. If you already uploaded a raw template, click ↻ Transform on its row to apply the same conversion in place.
  1. Open the relevant Security Assessment XXX … (Template).docx in Microsoft Word. Editing in LibreOffice can re-wrap split runs and break Jinja tags that span runs — Word is safest.
  2. Replace any literal text or table cell with the placeholder below. Keep the {{ … }} / {% … %} syntax intact — break a curly-brace pair across runs and the renderer raises a Jinja syntax error.
  3. Save the file under its original name and drop it back into the same report-templates/ folder (or upload via the "Replace .docx" action on the Report Templates tab).
  4. Click ↻ Regenerate from VibeDocs sources above. The transformer rewrites the canonical <code>_template.docx files used at render time. No restart needed.
  5. Generate a test report and validate the output. If something renders as literal Jinja text, run /api/templates/diagnose-defaults to see which source each template resolved from.

Scanning tools table loop

To render the Vulnerability Scanning Tools Used table, add a {%tr for … %} row loop inside the table in your Word template:

| Name | Version | Last Update Performed |
| {%tr for t in scanning_tools %}
| {{ t.tool_name }} | {{ t.version }} | {{ t.last_update }} |
| {%tr endfor %}

Each row the consultant adds in Report Details becomes one table row. An empty list renders zero data rows (header row only).

Per-finding loop

Findings render inside a paragraph-level loop. Wrap the section you want repeated with these two tags:

{%p for f in findings %}
  Heading 2:  {{ f.title }}

  Risk-rating table (one row per finding):
    | {{ f.severity }} | {{ f.cvss_score }} | {{ f.cvss_vector }} | {{ f.owasp_category }} | {{ f.cwe }} | {{ f.severity }} |

  Affected Asset:      {{ f.affected_asset }}
  Observations:        {{ f.description }}
  Steps to Reproduce:  {{ f.poc_steps }}
  Implications:        {{ f.impact }}
  Recommendations:     {{ f.remediation }}
  References:          {{ f.references }}
  Management Comments: {{ f.client_statement }}
  Status:              {{ f.status }}
  Follow-Up:           {{ f.retest_notes }}

  Screenshots (paragraph-level loop — one image + caption per iteration):
  {%p for img in f.screenshot_objs %}
  {{ img }}
  Figure {{ loop.index }}: {{ f.title }}
  {%p endfor %}

  Retest / follow-up screenshots:
  {%p for img in f.retest_objs %}
  {{ img }}
  Figure {{ loop.index }}-retest: {{ f.title }}
  {%p endfor %}
{%p endfor %}

Embedding screenshots inline in rich-text fields

Consultants can reference uploaded screenshots from inside the Observations, Steps to Reproduce, Implications, Recommendations, References, Follow-up, and Client Statement editors by typing [Screenshot N] on a new line, where N is the 1-based index in the finding's screenshots upload list. The renderer replaces the token with the matching image at generate time: 

1. Open the login page.
[Screenshot 1]
2. Enter `' OR 1=1 --` in the username field.
[Screenshot 2]
3. Submit and observe the bypass.

Out-of-range references ([Screenshot 99] when only two are uploaded) keep their literal text in the deliverable so the consultant sees the dangling reference. The dedicated Screenshots SubHeading section of the finding still renders every uploaded screenshot in order — inline references are an add-on for cases where you want a specific image to land next to the step that produced it.

Pitfalls & gotchas

  • Don't manually type the curly braces in a tight loop. Word splits them across runs and the Jinja parser fails. Paste the whole placeholder in one go, or use Find & Replace.
  • Tables loop with {%tr ... %} (row-level), paragraphs with {%p ... %} (paragraph-level). Using the wrong scope produces a duplicated table outside the intended scope.
  • Inline images via {{ f.screenshot_objs }} need to live inside a paragraph context — don't drop them inside a table cell unless the cell already has a paragraph.
  • Plain {{ x }} vs {% raw %}{{ x }}{% endraw %}: only use {% raw %} if you want the literal curly braces to render. The system strips stray raw blocks at render time, but it's cleaner not to add them.
  • Stale runs. After heavy editing in Word, save → close → reopen the file. Word often leaves zombie runs behind that break Jinja parsing only on save.
